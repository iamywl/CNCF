# Gemini CLI 토큰 최적화 및 비용 절감 가이드

Gemini CLI는 매 턴마다 전체 대화 이력을 모델에 전송하므로, 대화가 길어질수록 토큰 소모량이 기하급수적으로 증가합니다. 이를 최적화하기 위한 시니어 엔지니어의 전략을 정리합니다.

## 1. 도구(Tool) 활용 최적화

### [1] `read_file` 범위 지정 (Surgical Reading)
수천 줄의 파일을 통째로 읽는 것은 토큰 낭비의 주범입니다.
*   **나쁜 예:** `read_file(file_path="large_file.go")`
*   **좋은 예:** `read_file(file_path="large_file.go", start_line=100, end_line=200)`
*   **팁:** `grep_search`로 먼저 위치를 찾고, 필요한 부분만 잘라서 읽으십시오.

### [2] `grep_search` 컨텍스트 활용
검색 후 파일을 다시 읽는 턴(Turn)을 줄이십시오.
*   **전략:** `grep_search` 호출 시 `context`, `before`, `after` 파라미터를 사용하여 검색 결과 주변의 코드를 한 번에 가져오면 `read_file` 호출 횟수를 줄일 수 있습니다.

### [3] 불필요한 파일 제외 (`.geminiignore`)
`node_modules`, `dist`, `.git`, 대규모 바이너리 파일 등이 검색 및 분석 대상에 포함되지 않도록 `.geminiignore` 파일을 설정하십시오.

## 2. 세션 및 컨텍스트 관리

### [1] Sub-Agent 활용 (Task Compression)
복잡하거나 반복적인 작업은 `generalist`나 `codebase_investigator`에게 위임하십시오.
*   **원리:** 서브 에이전트의 수십 번의 턴은 메인 세션에 **단 한 줄의 요약**으로 기록됩니다. 메인 세션의 'Permanent History'를 가볍게 유지하는 가장 강력한 방법입니다.

### [2] 정기적인 세션 초기화 및 컴팩트화
*   **`/compact` 명령어:** 현재까지의 대화 이력을 요약본으로 압축하고 상세 이력을 지워 컨텍스트를 비웁니다.
*   **작업 단위 세션 분리:** 프로젝트 분석이 끝나고 구현 단계로 넘어갈 때, 혹은 새로운 프로젝트를 시작할 때는 새 세션을 시작하는 것이 비용 면에서 가장 유리합니다.

## 3. 전략적 프롬프팅 (Instruction Precision)

### [1] 구체적인 경로 명시
"어디선가 X 로직을 찾아줘"라는 모호한 요청은 Gemini가 여러 번의 `glob`과 `grep`을 수행하게 만들어 턴을 낭비합니다. 알고 있는 경로가 있다면 명시하십시오.

### [2] '리서치'와 '실행'의 분리
*   먼저 "분석만 해줘(Inquiry)"라고 시킨 뒤, 결과가 만족스러우면 "이제 반영해줘(Directive)"라고 지시하십시오. 시행착오에 따른 코드 재작성 토큰을 아낄 수 있습니다.

## 4. 실전 체크리스트

4.  [ ] 반복 작업은 `generalist`에게 던졌는가?
5.  [ ] `.geminiignore`가 제대로 설정되어 있는가?
