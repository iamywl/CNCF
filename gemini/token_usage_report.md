# Gemini CLI 토큰 소모량 추산 및 비용 최적화 보고서

본 보고서는 Gemini CLI를 활용한 CNCF 오픈소스 분석 작업 시 발생하는 토큰 소모량을 유형별로 정밀 추산하고, 이를 관리하기 위한 기술적 가이드를 제공한다.

## 1. 핵심 요약 (Executive Summary)
Gemini CLI의 토큰 소모는 시스템 프롬프트(약 5k)와 대화 이력의 누적에 의해 결정된다. 단순 질의는 20k 내외에서 해결 가능하나, 전체 코드베이스 분석 및 문서 생성 단계에서는 프로젝트당 최대 900k 이상의 토큰이 소모될 수 있다. 따라서 세션 초기화와 서브 에이전트 활용을 통한 컨텍스트 관리가 필수적이다.

## 2. 토큰 소모 결정 요인 분석
토큰 소모량은 크게 고정 소모량과 가변 소모량으로 구분된다.

*   **고정 소모량 (Fixed Cost):** 매 턴마다 전송되는 시스템 지침(System Instructions)과 도구 정의(Tool Definitions)로, 약 3,000 ~ 5,000 토큰이 기본적으로 소모된다.
*   **가변 소모량 (Variable Cost):**
    *   **대화 이력 (History):** 이전 대화 내용이 매번 다시 전송되므로 대화가 길어질수록 소모량이 기하급수적으로 증가한다.
    *   **도구 출력 (Tool Output):** `ls -R`이나 `read_file`을 통해 가져오는 코드베이스의 물리적 크기가 가장 큰 비중을 차지한다.
    *   **생성 텍스트 (Completion):** 모델이 작성하는 문서와 코드의 길이가 길수록 출력 토큰 비용이 발생한다.

## 3. 작업 유형별 정밀 추산 데이터
실제 오픈소스 프로젝트(Go 언어 기준) 분석 시 발생하는 평균 소모량이다.

| 작업 카테고리 | 상세 작업 내용 | 예상 턴 수 | 추산 토큰 (Range) |
| :--- | :--- | :---: | :--- |
| **코드 탐색 (Search)** | 특정 심볼 검색, 디렉토리 구조 파악 | 3~5회 | 30k ~ 60k |
| **심층 리서치 (Research)** | 핵심 로직 추적, 인터페이스 관계 분석 | 10~15회 | 150k ~ 300k |
| **코드 수정 (Refactor)** | 버그 수정, 테스트 작성 및 검증 | 10~20회 | 200k ~ 500k |
| **전체 분석 (Investigation)** | 아키텍처 매핑 및 의존성 그래프 작성 | 5~10회 | 100k ~ 250k |
| **신규 프로젝트 초기화** | 컨텍스트 패키징 및 문서 초안 생성 | 15~25회 | 500k ~ 900k |

## 4. 비용 절감을 위한 기술적 권장 사항
토큰 효율성을 극대화하기 위해 다음과 같은 기법을 반드시 적용해야 한다.

1.  **부분 읽기 (Surgical Read):** `read_file` 사용 시 파일 전체를 읽지 않고 `start_line`과 `end_line`을 지정하여 필요한 코드 블록만 로드한다. 이는 전체 토큰 소모를 80% 이상 절감할 수 있는 가장 효과적인 방법이다.
2.  **컨텍스트 압축 (History Management):** 대화가 20턴을 넘어갈 경우 `/compact` 명령어를 사용하여 이전 대화를 요약하고 히스토리를 정리한다.
3.  **병렬 검색 (Parallel Search):** 독립적인 검색 작업은 여러 개의 `grep_search`를 한 번에 호출하여 턴 수를 줄인다.
4.  **서브 에이전트 격리 (Sub-Agent Isolation):** 복잡한 반복 작업은 `generalist` 에이전트에게 위임한다. 서브 에이전트의 상세 작업 로그는 메인 세션에 요약본으로만 남기 때문에 메인 세션의 컨텍스트 비대화를 방지할 수 있다.

본 가이드를 준수함으로써 동일한 비용 대비 약 2~3배 더 많은 분석 작업을 수행할 수 있다.
