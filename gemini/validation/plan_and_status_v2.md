# CNCF 오픈소스 심화 분석 및 기술 검증 통합 마스터 플랜 v2

본 문서는 CNCF 오픈소스 프로젝트의 본질을 꿰뚫는 **'심화 분석(Deep-Dive)'**과 그 결과물의 신뢰성을 보장하는 **'기술 검증(Verification)'**을 하나로 통합한 최상위 작업 지침서이다.

## 1. 핵심 철학 (Mastery Philosophy)

*   **개념 완결성 (Concept Completeness):** 단순한 기능 나열이 아닌, 시스템의 모든 핵심 서브시스템과 데이터 흐름을 빠짐없이 분석한다. PoC의 수량(18개 등)에 얽매이지 않고, **모든 핵심 개념을 100% 커버**할 때까지 분석을 멈추지 않는다.
*   **실증적 검증 (Empirical Validation):** "동작하지 않는 지식은 죽은 지식이다." 모든 분석 내용은 실제 소스코드와 대조되어야 하며, 모든 PoC 코드는 로컬 환경에서 **성공적으로 실행(Go Run)**되어야 한다.
*   **맥락적 이해 (Contextual Understanding):** 현재의 코드가 왜(Why) 이렇게 설계되었는지를 이해하기 위해 **역사적 배경(PR/Issue)**과 **테스트 코드의 의도**를 추적한다.

## 2. 통합 분석 및 검증 워크플로우 (6단계 프로세스)

각 프로젝트는 다음의 엄격한 6단계를 거쳐 'Validated' 상태로 진입한다.

### [1단계] 아키텍처 및 역사적 맥락 분석 (Research & Context)
*   **작업:** 진입점 식별, `git blame` 및 PR 추적을 통한 설계 의도(Design Trade-off) 파악.
*   **검증 도구:** Gemini Pro (1M+ 컨텍스트 활용).
*   **산출물:** `claude_context_package.md` (클로드에게 전달할 정제된 리서치 결과).

### [2단계] 테스트 코드 기반 역공학 (Test-driven Analysis)
*   **작업:** 본체 코드보다 `*_test.go`를 먼저 분석하여 함수의 경계 조건(Edge Case)과 정상 작동 시나리오 파악.
*   **목표:** 추상화 레이어 뒤에 숨겨진 실제 구현 책임(Responsibility) 식별.

### [3단계] 전방위 문서화 및 PoC 설계 (Drafting & Designing)
*   **작업:** 500줄 이상의 심화 분석 문서(10~12개) 작성 및 해당 문서와 1:1 매핑되는 PoC 설계.
*   **PoC 전략:** 서브시스템당 최소 1개 이상의 PoC를 제작하며, 수량 제한 없이 **개념 커버리지**를 우선함.

### [4단계] 실행 기반 PoC 구현 (Implementation)
*   **작업:** Go 표준 라이브러리만을 활용하여 핵심 알고리즘 재현.
*   **품질:** `go run main.go`로 즉시 실행 가능해야 하며, 내부 로직을 상세히 설명하는 주석 포함.

### [5단계] 정적/동적 무결성 검증 (Technical Audit)
*   **작업 1 (Static):** 문서 내 언급된 파일 경로, 함수명, 구조체명이 실제 코드와 일치하는지 전수 조사.
*   **작업 2 (Dynamic):** PoC 코드를 직접 실행하여 결과값이 예상과 일치하는지 확인.
*   **검증 도구:** Gemini CLI (`ls`, `grep`, `run_shell_command`).

### [6단계] 최종 승인 및 상태 업데이트 (Quality Gate)
*   **기준:** 모든 검증 항목(Path, Symbol, Runtime) 통과 시에만 **`Validated`** 상태로 전이.

## 3. 검증 항목 상세 체크리스트 (Verification Checklist)

| 구분 | 검증 항목 | 세부 확인 내용 |
| :--- | :--- | :--- |
| **정확성 (Accuracy)** | 파일 경로(Path) | 문서 내 인용된 모든 파일이 실제 경로에 존재하는가? |
| | 심볼(Symbol) | 함수, 인터페이스, 구조체명이 오타 없이 기재되었는가? |
| **재현성 (Reproducibility)** | PoC 실행 | `go run` 시 컴파일 에러나 런타임 패닉이 없는가? |
| | 외부 의존성 | 표준 라이브러리 외의 외부 패키지를 사용하지 않았는가? |
| **깊이 (Depth)** | 개념 커버리지 | 프로젝트의 모든 핵심 서브시스템이 PoC로 구현되었는가? |
| | 설계 근거 | "왜 이렇게 설계했는가"에 대한 PR/Issue 근거가 포함되었는가? |
