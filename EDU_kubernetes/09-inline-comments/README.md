# 09. 인라인 주석 (Inline Comments) — 효과적인 주석 작성법

## 개념

인라인 주석은 코드 바로 옆이나 위에 작성하는 설명입니다.
**'어떻게(How)'가 아닌 '왜(Why)'** 를 설명할 때 가치가 있습니다.

## 왜 필요한가?

- 비즈니스 로직상 **특이한 처리**가 필요한 경우 이유를 남김
- 성능 최적화로 인해 **직관적이지 않은 코드**를 작성했을 때
- **외부 제약**(API 한계, 브라우저 버그 등)으로 인한 우회 코드

## 좋은 주석 vs 나쁜 주석

### 나쁜 주석 (코드를 읽으면 아는 것)
```javascript
// i를 1 증가시킨다
i++;

// 배열을 정렬한다
arr.sort();

// 사용자를 가져온다
const user = getUser(id);
```

### 좋은 주석 (코드만으로는 알 수 없는 것)
```javascript
// PG사 API가 간헐적으로 타임아웃을 반환하므로 최대 3회 재시도
const MAX_PAYMENT_RETRIES = 3;

// Safari 14 이하에서 Intl.DateTimeFormat이 한국어를 지원하지 않아
// 수동 포맷팅으로 우회
function formatDateKorean(date) { ... }

// 이 쿼리는 인덱스(idx_order_date)를 타도록 ORDER BY를 유지해야 함
// 제거 시 풀 테이블 스캔으로 응답 시간 10배 증가 (2024-03 성능 테스트 결과)
const orders = await db.query('SELECT ... ORDER BY order_date');
```

## 주석 유형 가이드

| 유형 | 언제 사용 | 예시 |
|------|-----------|------|
| **WHY 주석** | 비직관적인 결정의 이유 | `// 퀵정렬 대신 힙정렬: 최악 O(n log n) 보장 필요` |
| **WARNING 주석** | 위험한 코드 영역 경고 | `// WARNING: 이 순서를 바꾸면 데드락 발생` |
| **TODO 주석** | 향후 개선 사항 | `// TODO: 캐시 도입 후 이 N+1 쿼리 제거` |
| **HACK 주석** | 우회/임시 코드 | `// HACK: API v2 출시 전까지 이 필드를 수동 매핑` |
| **REF 주석** | 외부 참조 링크 | `// REF: https://bugs.chromium.org/p/...` |

## 학습 포인트

1. **WHY > HOW**: 코드가 '어떻게'를 보여주니, 주석은 '왜'를 설명
2. **날짜/맥락 포함**: 언제, 왜 이 결정을 했는지 기록
3. **TODO/HACK 태그**: 임시 코드임을 명시하여 기술 부채 추적 가능
4. **주석도 유지보수**: 코드가 바뀌면 주석도 업데이트 (거짓 주석이 최악)
