# PoC-07: 인터셉터 체이닝

## 개념

gRPC의 인터셉터(Interceptor) 체이닝 메커니즘을 시뮬레이션한다.

```
요청 ──▶ 인터셉터A ──▶ 인터셉터B ──▶ 인터셉터C ──▶ 핸들러
응답 ◀── 인터셉터A ◀── 인터셉터B ◀── 인터셉터C ◀── 핸들러

체이닝 구조 (양파 모델):
┌─────────────────────────────────────────┐
│ 로깅 인터셉터 (진입 로그)                  │
│  ┌──────────────────────────────────┐   │
│  │ 메트릭 인터셉터 (카운터 증가)       │   │
│  │  ┌───────────────────────────┐   │   │
│  │  │ 인증 인터셉터 (토큰 검증)    │   │   │
│  │  │  ┌────────────────────┐   │   │   │
│  │  │  │ 실제 핸들러 (비즈니스) │   │   │   │
│  │  │  └────────────────────┘   │   │   │
│  │  └───────────────────────────┘   │   │
│  └──────────────────────────────────┘   │
│ 로깅 인터셉터 (응답 로그)                  │
└─────────────────────────────────────────┘
```

## 시뮬레이션하는 gRPC 구조

| 구조체/함수 | 실제 위치 | 역할 |
|------------|----------|------|
| `UnaryServerInterceptor` | `interceptor.go:87` | 서버 Unary 인터셉터 |
| `UnaryClientInterceptor` | `interceptor.go:43` | 클라이언트 Unary 인터셉터 |
| `UnaryHandler` | `interceptor.go` | 실제 핸들러 함수 |
| `chainUnaryServerInterceptors` | `server.go` | 서버 인터셉터 체이닝 |
| `chainUnaryClientInterceptors` | `clientconn.go` | 클라이언트 인터셉터 체이닝 |

## 실행 방법

```bash
cd poc-07-interceptor

```

## 예상 출력

```
=== 인터셉터 체이닝 시뮬레이션 ===

── 1. 서버 인터셉터 체인 (로깅 → 메트릭 → 인증 → 핸들러) ──

  [케이스 1] 인증 성공:
    [서버/로깅] ▶ 요청: method=/greeter.Greeter/SayHello, req=World
    [서버/메트릭] 요청 카운트: 1
    [서버/인증] ✓ 인증 성공
    [서버/로깅] ◀ 응답: Echo: World (소요: ...)
  결과: Echo: World

  [케이스 2] 인증 실패 (요청 거부):
    [서버/로깅] ▶ 요청: ...
    [서버/메트릭] 요청 카운트: 2
    [서버/인증] ✗ 인증 실패
    [서버/로깅] ◀ 에러: 인증 실패
  결과: 에러=인증 실패: 유효하지 않은 토큰

── 5. 실행 순서 확인 ──
    [인터셉터 A] 진입 ▶
    [인터셉터 B] 진입 ▶
    [인터셉터 C] 진입 ▶
    [핸들러] 실행
    [인터셉터 C] 복귀 ◀
    [인터셉터 B] 복귀 ◀
    [인터셉터 A] 복귀 ◀

=== 시뮬레이션 완료 ===
```

## 핵심 포인트

1. **양파 모델**: 인터셉터는 핸들러를 감싸는 미들웨어 패턴 (진입 → 내부 → 복귀)
2. **체이닝**: 역순 래핑으로 인터셉터 순서 보장 (첫 번째 등록 = 가장 바깥)
3. **요청 거부**: 인터셉터가 handler를 호출하지 않으면 요청이 거부됨 (인증 실패 등)
4. **클라이언트 + 서버**: 양쪽 모두 독립적인 인터셉터 체인을 가짐
5. **실용 패턴**: 로깅, 인증, 메트릭, 재시도, 타임아웃, 패닉 복구
