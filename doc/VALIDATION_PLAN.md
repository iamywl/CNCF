# EDU 검증 및 보강 마스터 플랜 (자동화 도구 전용)

> 작성일: 2026-03-07
> 대상: 16개 CNCF 프로젝트 EDU (전체 완료 상태)
> 실행 환경: 자동화 환경 + 자동화 도구 CLI
> 작성 배경: 기존 EDU의 신뢰성 확보 및 커버리지 갭 보완

---

## 0. 왜 검증이 필요한가

| 리스크 | 설명 |
|--------|------|
| **환각(Hallucination)** | 소스코드에 존재하지 않는 파일 경로, 함수명, 구조체명이 문서에 인용되었을 가능성 |
| **피상적 PoC** | 핵심 알고리즘의 본질이 아닌 표면적 모방에 그친 PoC가 있을 가능성 |
| **커버리지 미달** | 프로젝트의 중요한 서브시스템이 심화문서/PoC에서 누락되었을 가능성 |
| **실행 불가** | `go run main.go` 실행 시 컴파일 에러 또는 런타임 패닉이 발생할 가능성 |
| **얕은 '왜(Why)'** | 설계 근거 없이 "이렇게 동작한다"만 서술한 문서가 있을 가능성 |

## 1. 검증 원칙

### Gemini 계획과의 차이점

| 항목 | Gemini v2 계획 | 자동화 도구 계획 |
|------|---------------|-----------------|
| 역할 분리 | Gemini=검증자, Claude=작성자 | **자동화 도구가 검증+보강 일체 수행** |
| 검증 방식 | CLI로 grep/ls 수동 확인 | **Grep/Glob/Read/Bash 도구로 자동화** |
| PoC 검증 | 별도 실행 환경 필요 | **Bash 도구로 `go run` 직접 실행** |
| 컨텍스트 | 1M 토큰 한번에 투입 | **필요한 파일만 선택적 Read, /compact로 관리** |
| 토큰 제약 | Gemini API 토큰 비용 | **자동화 환경 = 무제한 메시지 (레이트 리밋만 존재)** |
| 일정 단위 | 주간 루틴 (월~토) | **프로젝트 단위 세션, 하루 복수 프로젝트 가능** |

### 3無 + 2有 원칙

1. **기능 전수 조사:** 프로젝트의 핵심 기능/서브시스템을 소스코드 기반으로 전부 목록화
2. **커버리지 매핑:** 기존 EDU(심화문서 + PoC)가 그 중 무엇을 다뤘고 무엇을 빠뜨렸는지 대조
3. **갭 명세:** 누락된 기능/서브시스템을 우선순위와 함께 명확히 리스트업
4. **보강 판단:** 갭이 크면 추가 문서/PoC 작성, 작으면 검증 완료 처리

## 2. 검증 프로세스 (2단계)

### [Phase 1] 기능/서브시스템 전수 조사 + 커버리지 매핑

프로젝트별로 다음을 수행한다:

**Step 1 — 프로젝트 전체 기능 목록 생성**
```
1. 소스코드 디렉토리 구조 분석 (tree -L 2 -d, cmd/, pkg/, internal/)
2. README.md, 공식 문서에서 핵심 기능/개념 추출
3. 주요 패키지별 역할 파악 (각 디렉토리가 담당하는 기능)
4. 프로젝트의 "핵심 기능 전체 목록" 확정
```

**Step 2 — 기존 EDU 커버리지 매핑**
```
1. 심화문서(07~18-*.md) 각각이 어떤 서브시스템/기능을 다루는지 목록화
2. PoC(poc-*/) 각각이 어떤 개념을 구현하는지 목록화
3. Step 1의 전체 목록 vs Step 2의 커버 목록 대조
```

**Step 3 — 갭 리포트 출력**
```
프로젝트: {이름}
전체 핵심 기능: N개
EDU 커버: M개 (XX%)
누락 목록:
  [P0-핵심] 기능A — 설명
  [P0-핵심] 기능B — 설명
  [P1-중요] 기능C — 설명
  [P2-선택] 기능D — 설명
```

**커버리지 판정:**
| 등급 | 기준 | 의미 |
|------|------|------|
| A (90%+) | P0 누락 0개 | EDU 충분, 보강 불필요 |
| B (70~89%) | P0 누락 1~2개 | 핵심 일부 보강 필요 |
| C (50~69%) | P0 누락 3개+ | 상당한 보강 필요 |
| D (<50%) | 절반 이상 미커버 | 재작성 수준 |

### [Phase 2] 보강 (사용자 판단 후 선택적 수행)

Phase 1의 갭 리포트를 사용자에게 제시한 뒤, 사용자가 보강 범위를 결정한다.

```
사용자 판단:
  - "이 프로젝트는 A등급이니 패스"
  - "P0 누락 2개만 추가해줘"
  - "전부 보강해줘"

보강 시 수행:
  - 누락 서브시스템 심화문서 추가 (소스코드 분석 → 500줄+ 문서)
  - 누락 개념 PoC 추가 (표준 라이브러리만, go run 실행 가능)
```

## 3. 프로젝트별 검증 우선순위

난이도와 중요도 기반으로 3개 그룹으로 나눈다.

### Group A — 핵심 (가장 먼저, 가장 꼼꼼하게)

| 프로젝트 | 이유 | 예상 리스크 |
|----------|------|------------|
| **kubernetes** | 가장 큰 프로젝트, PoC 26개로 오류 확률 높음 | 방대한 소스 → 환각 경로 가능성 ↑ |
| **cilium** | eBPF/커널 레벨 → 검증 난이도 최고 | C 코드 참조 정확성 |
| **istio** | 서비스 메시 복잡도 높음 | Envoy/xDS 심화 커버리지 |

### Group B — 중요 (핵심 경로 위주 검증)

| 프로젝트 | 이유 | 예상 리스크 |
|----------|------|------------|
| **containerd** | 런타임 핵심, OCI 스펙 정합성 | 스냅샷/이미지 관련 경로 |
| **argo-cd** | GitOps 핵심 흐름 | Sync/Reconciliation 로직 |
| **kafka** | Java/Scala 혼용 → 환각 가능성 | JVM 코드 참조 정확성 |
| **terraform** | HCL/Provider 아키텍처 복잡 | 플러그인 시스템 커버리지 |
| **grafana** | Go/TS 혼용, 프론트엔드 영역 | 대시보드/패널 서브시스템 |

### Group C — 보통 (경량 검증)

| 프로젝트 | 이유 |
|----------|------|
| **helm** | 상대적으로 단순한 구조 |
| **grpc-go** | 단일 언어, 명확한 구조 |
| **alertmanager** | 소규모 프로젝트 |
| **hubble** | Cilium 하위 프로젝트 |
| **jaeger** | 단일 도메인 (트레이싱) |
| **loki** | Grafana 계열, 구조 유사 |
| **tart** | Swift, 소규모 |
| **jenkins** | Java 단일 언어 |

## 4. 세션 운영 전략 (자동화 환경 최적화)

### 자동화 환경 특성 고려

| 항목 | 값 | 전략 |
|------|-----|------|
| 메시지 제한 | 무제한 (레이트 리밋 있음) | 하루에 여러 세션 운영 가능 |
| 컨텍스트 윈도우 | 200K 토큰 | `/compact` 적극 활용, 프로젝트별 세션 분리 |
| 모델 | Opus 4.6 | 심층 분석에 최적, 환각률 낮음 |
| 도구 | Read/Grep/Glob/Bash 전부 사용 | 검증 자동화 가능 |

### 세션 분할 전략

```
한 프로젝트 검증 = 1~2 세션

세션 1: Phase 1 (커버리지 분석)
  - 소스코드 디렉토리/README 분석 → 전체 기능 목록 확정
  - 기존 EDU 심화문서 + PoC 커버리지 매핑
  - 갭 리포트 출력 → 사용자에게 보고
  - /compact 후 다음 세션

세션 2: Phase 2 (보강) — 사용자가 요청할 때만
  - 사용자가 지정한 누락 항목에 대해 심화문서 + PoC 추가
```

### 병렬 처리 활용

```
Agent 도구를 활용하여 독립적 커버리지 분석을 병렬 수행:

  Agent 1: 소스코드 디렉토리 구조 분석 → 전체 기능 목록 생성
  Agent 2: 기존 심화문서(07~18) 내용 요약 → 커버 목록 생성
  Agent 3: 기존 PoC(poc-*/) 내용 요약 → 커버 목록 생성

  → 3개 결과를 합쳐서 갭 분석
```

### 일일 목표

| 그룹 | 프로젝트당 예상 시간 | 하루 처리량 |
|------|---------------------|------------|
| Group A | 3~4세션 | 1 프로젝트/일 |
| Group B | 2~3세션 | 1~2 프로젝트/일 |
| Group C | 1~2세션 | 2~3 프로젝트/일 |

**총 예상 기간:** 10~14일 (16개 프로젝트 전수 검증)
